Writing Style Guide — SHUKE‑LABS blog

Voice and intent
- First person, accountable: these are the author's decisions and tradeoffs.
- Plain and direct: avoid jargon and hype. Keep a modest, slightly wry tone.

Structure
- Lead with the point. First paragraph should state the takeaway.
- Use short sections. Each section covers one idea.
- Follow problem → attempted approaches → what worked → concrete rule.

Sentences and grammar
- Prefer active voice and short-to-medium sentences.
- One idea per sentence. Avoid dense multi-clause sentences.

Evidence and specificity
- Use concrete artifacts: dates, PR counts, labels, file names.
- When asserting behavior, provide the minimal data or pointer.

Tone limits
- No clickbait. No grandiose claims.
- No fluff: delete sentences that don't add actionable or explanatory value.

Practical rules for translators/agents
- Keep frontmatter meta: title, description, pubDate, lang, tags.
- Preserve commits/dates when they are cited as evidence.
- Prefer human-like phrasing over literal word-for-word translation when preserving tone.

How to handle uncertainty
- If unsure about a fact, mark it as "(approx.)" or leave a TODO comment for the author to confirm.

Length
- Blog posts: 400–1,500 words. Theses: 800–2,000 words. Keep paragraphs short.

Examples
- Good lead: "I built X because Y was wasting my time." (Direct, personal, outcome-oriented.)
- Bad lead: "Recent advances in X suggest..." (Abstract, impersonal.)

End with where to find source material (commits, issues, chat logs) so readers can verify claims.
